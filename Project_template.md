## Задание 5. Запуск и демонстрация работы бота.

Для защиты RAG пайплайна будем использовать библиотеку llm-guard.
Этот модуль решает задачи фильтрации pre-фильтр чанков и post-фильтр ответа, без внешнего API.

### Input Scanners

* PromptInjection — основной защитный сканер для RAG-пайплайна. Использует ML-модель 
(дообученный классификатор), которая определяет, пытается ли текст чанка перехватить управление моделью. 
В отличие от regex-подхода, он устойчив к вариациям формулировок: "ignore instructions",
"IGNORE INSTRUCTIONS", "igNorE aLL iNsTruCtIoNs" — всё это будет поймано. 
Для RAG это критично, потому что злоумышленник может загрузить документ с инжекцией в базу знаний, 
и без этого сканера модель выполнит команду из документа как будто это инструкция пользователя.

### Output Scanners

* NoRefusal — детектирует ситуацию, когда модель "сломалась" под воздействием инжекции и 
начала отказываться от своей роли: фразы типа "As an AI I cannot...", "I don't have restrictions...", 
"My previous instructions said...". Это пост-проверка того, что Input Scanners не поймали на уровне чанков. 
Без него успешная инжекция остаётся незамеченной — пайплайн вернёт скомпрометированный ответ как валидный.
* Sensitive — детектирует утечку чувствительных данных в ответе модели: ключи API, токены, персональные данные, 
внутренние системные инструкции. Актуален потому, что один из классических векторов атаки 
через RAG — заставить модель воспроизвести содержимое system prompt или другие чанки из базы, 
которые не должны попадать к пользователю.

Возможные проблемы:

1. Невалидный чанк
Возникает на этапе подготовки датасета, когда retriever вернул None, пустую строку или несовместимый 
тип в поле contexts. Ragas падает с ValidationError до старта оценки. Необходимо валидировать схему датасета 
до передачи в Ragas — это дешевле, чем отлавливать ошибку потом. 

2. Пустой индекс
Retriever получает пустой список — либо индекс физически пуст, либо порог схожести отсекает все результаты, 
либо модель эмбеддингов при запросе не совпадает с той, которой индексировались документы. 
Последнее самое коварное: система не падает, просто возвращает бессмысленные результаты. 

По полученным результатам можно сделать вывод, что ответы генерируются релевантные. Но для более полного внедрения
концепций CoT и few-shot необходимо использование более мощной LLM. Практика показала, что текущая LLM не справляется
с большим запросом пользователя (особенно когда в нем есть длинные примеры).

Так же было бы неплохо расширить выборку вопросов для тестирования, 10 - это очень мало. 
Уместно добавить валидацию перед подсчетом метрик в Ragas и расширить сам набор метрик.

Улучшения можно добавить и в нашу система RAG api. Расширить валидацию и добавить более детальное логирование
в случаях когда контекст по запросу пользователя найти не удалось, либо система отбросила данные по результатам
проверки безопасности. Необходимо настроить работу с метаданными.
