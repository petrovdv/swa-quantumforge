
## Задание 1. Исследование моделей и инфраструктуры

Согласно требованиям текущая аудитория планируемого решения - это внутренняя команда (с оговоркой, что в будущем возможно будут предлагать и внешним клиентам). Текущие роли пользователей базы знаний: разработчики, саппорт, менеджеры, аналитики и новые сотрудники. Продакшн среда компании развернута в AWS облаке, большинство используемых инструментов - облачные SAAS решения. Поскольку большинство документов предназначены для внутренней аудитории (скорее всего, содержат конфиденциальные данные), то будем рассматривать локальное разворачивание модели LLM. В будущем, при необходимости, рассмотрим миграцию в облако.

**1. Сравните LLM-модели** (локальные Hugging Face vs облачные OpenAI / YandexGPT):

|                                    | Облачные модели (OpenAI, YandexGPT)                                                                            | Современные open-source модели (LLaMA, Qwen, Mistral)                                                                                                                |
| ---------------------------------- | -------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Качество ответов                   | Лучшие на рынки, постоянно обновляются и совершенствуются                                                      | Качество зависит от размеров модели, приближаются к облачным, но уступают им                                                                                         |
| Скорость работы                    | Есть сетевая задержка, но хорошо масштабируется и более производительная за счёт доступности облачных ресурсов | Нет сетевой задержки, но зависит от доступных CPU/GPU ресурсов                                                                                                       |
| Стоимость владения и использования | Оплата за токены, нет капитальных затрат, платишь только за использование                                      | Нужны капитальные затраты на покупку вычислительного оборудования и в дальнейшем эксплуатационные затраты на поддержку его работоспособности и оплату электроэнергии |
| Удобство и простота развертывания  | Быстрый старт, удобно на старте проекта для проверки гипотезы. Из минусов - привязка к одному вендору.         | Полный контроль и возможность настройки под себя, контроль за потоками данных. Из минусов - требуются инженерные ресурсы для обслуживания и настройки.               |

Для текущей задачи будем работать с локальной LLM (Qwen/Qwen3.5-2B), учитывая требования к конфиденциальности данных и то, что база знаний нужна для внутренней команды. В контексте поставленной задачи (база знаний) - качества ответов локальных моделей нам будет достаточно, а удобство их настройки будет плюсом. В дальнейшем после введения системы в эксплуатацию, можно рассмотреть переход к гибридной архитектуре.

**2. Сравните модели эмбеддингов** (локальные Sentence-Transformers vs облачные OpenAI Embeddings):

|                                    | локальные Sentence-Transformers (all-MiniLM-L6-v2, bge-small) | облачные OpenAI Embeddings                                          |
| ---------------------------------- | ------------------------------------------------------------- | ------------------------------------------------------------------- |
| скорость создания индекса          | Одна из самых быстрых                                         | Есть сетевые задержки, возможны ограничения на частоту запросов     |
| качество поиска                    | Хорошее качество, достаточное для базы знаний                 | Более высокое качество                                              |
| стоимость владения и использования | Бесплатно                                                     | Оплата за токены, стоимость становится заметной при больших объемах |

В контексте текущей задачи будем использовать all-MiniLM-L6-v2.

**3. Сравните векторные базы ChromaDB и FAISS:**

|                                          | ChromaDB                                                                                     | FAISS                                                                               |
| ---------------------------------------- | -------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------- |
| скорость поиска                          | Немного уступает FAISS                                                                       | Очень высокая, максимально оптимизирован (С++), подходит для high-load систем       |
| скорость индексации                      | На малых и средних данных - быстрая, на больших может уступать FAISS                         | Может занимать время на больших данных, оптимизирована для масштабирования          |
| сложность внедрения и поддержки          | Высокоуровневый API (REST из коробки), встроенная поддержка метаданных, хорошая документация | Много технических нюансов, нет API для хранения метаданных, нет REST api из коробки |
| удобство в работе                        | Быстрый запуск, все функции "из коробки"                                                     | Гибкие настройки индексов, требует самостоятельной организации хранения метаданных  |
| стоимость владения (учёт инфраструктуры) | Бесплатно, затраты на инфраструктуру (DevOps и разработка)                                   | Бесплатно, затраты на инфраструктуру (DevOps и разработка)                          |

ChromaDB обычно дешевле и проще в поддержке для малых и средних нагрузок. Для миллиардов векторов и максимальной производительности FAISS будет более затратным, но мощнее. Текущую задачу попробуем решить с использованием ChromaDB.

ChromaDB — «Pythonic» вариант: всё живёт в процессе, идеально интегрируется с LangChain, прозрачен для быстрых PoC. Однако в тот момент, когда коллекция состоит уже из десятков миллионов векторов, придётся мигрировать куда-то ещё.

В контексте нашей задачи можно двигаться в реализации по этапам от первого варианта к третьему. В качестве учебного материала реализуем прототип по варианту 1. Хорошоее эксплуатационное решение для внедрения - вариант 2. Лучший вариант, к которому можем прийти в дальнейшем - вариант 3. При этом учитывая, что у наше приложение не имеет высоких требований по производительности (аудитория - внутренняя команда), а основной паттерн использования - база знаний, можно обойтись без GPU.
